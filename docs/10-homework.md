# Домашнее задание

## Задание 1. Защита от невалидных `page` и `perPage` в URL

На занятии мы добавили базовую защиту: если значение из URL «не наше» — используем дефолт. Сейчас на странице `/places` это уже работает для `page` и `perPage`, но визуально пользователь не понимает, что что-то пошло не так.

Доработайте поведение страницы так, чтобы при невалидном `page` или `perPage` в URL вы тихо «исправляли» адрес:

1. Если `page` в URL невалидный (`?page=foo`, `?page=-1`, `?page=0`) — удалите параметр `page` из URL через `setSearchParams`. Используйте `{ replace: true }`, чтобы не засорять историю.
2. То же самое для `perPage`: если значение не входит в список `[3, 5, 10]` — удалите параметр.
3. Сделайте это в `useEffect`, который зависит от `pageRaw` и `perPageRaw` (а не от уже валидированных `page`/`perPage`). Это разные значения: первые приходят из URL, вторые — после защиты.

Проверьте:

- открыли вручную `/places?page=foo` → URL стал `/places`, страница показывает 1-ю;
- открыли `/places?perPage=999` → URL стал `/places`, perPage снова 5;
- адресная строка не моргает в бесконечном цикле (важно не передать `page`/`perPage` в зависимости — только `pageRaw`/`perPageRaw`).

## Задание 2. Состояние «нет результата» с подсказкой

Сейчас при пустом ответе мы показываем «Ничего не найдено по выбранным фильтрам». Сделайте этот блок чуть полезнее.

1. Если фильтров нет, текст «Список мест пока пуст.» — оставьте как есть.
2. Если есть хоть один фильтр (`search`, `status`, `sort`), показывайте две строки:
   - заголовок «Ничего не найдено по выбранным фильтрам»;
   - кнопку «Сбросить фильтры» — ту же логику, что и в основной панели.
3. Дополнительно (по желанию): если активен только `search`, поясните пользователю, что поиск идёт по названию (`title`), а не по описанию.

Проверьте:

- ввели в поиск явно несуществующее слово — блок появился, кнопка работает;
- после сброса URL стал `/places`, появились все места.

## Задание 3. Серверная сортировка «новые сначала» (необязательное, сложное)

Добавьте на страницу `/places` ещё один режим сортировки — «новые сначала» (по `visitedYear` по убыванию). json-server v1 beta поддерживает desc через префикс `-`.

1. Расширьте список `ALLOWED_SORTS` новым значением, например `-visitedYear`.
2. В `<select>` для сортировки добавьте пункт «Новые сначала» с этим значением.
3. Убедитесь, что значение из URL валидируется: невалидное → «по умолчанию».
4. В API-слое (`fetchPlacesPage`) ничего менять не нужно — он просто кладёт `?_sort=${params.sort}`, а минус json-server уже умеет.
5. На пустых `visitedYear` (например, `wishlist` / `planned` без года) места уйдут вниз — это нормальное поведение json-server. В `guide.md` коротко зафиксируйте, как сортировка ведёт себя с `undefined` значениями.

Подсказки:

- проверьте Network: после выбора «Новые сначала» запрос идёт с `_sort=-visitedYear`;
- помните про сброс `page` — он у вас уже автоматический в `updateParam`, потому что `sort` в списке `FILTER_KEYS_THAT_RESET_PAGE`;
- не вводите для `order` отдельный параметр в URL — на этом задании достаточно «совмещённого» поля `sort`.

Проверьте:

- выбрали «Новые сначала» → запрос пошёл с `_sort=-visitedYear`, посещённые места отсортированы от более новых к более старым;
- скопировали URL `/places?sort=-visitedYear` → открыли в новой вкладке — тот же порядок;
- ввели `?sort=hacker` — страница не упала, сортировка трактуется как «по умолчанию».
