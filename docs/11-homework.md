# Домашнее задание

## Задание 1. Свой `useFavoritesQuery` поверх существующих хуков

На занятии мы вынесли в `src/features/places/` хуки `usePlacesQuery`, `usePlaceQuery` и три мутации. Список «Избранное» (`/favorites`) пока работает по-старому: читает ids из `localStorage` и где-то рядом тащит данные руками.

Сделайте маленький custom hook, который собирает «избранные места» из существующего кеша TanStack Query:

1. Создайте файл `src/features/places/useFavoritePlaces.ts`.
2. Внутри возьмите ids избранного через `useLocalStorage`.
3. Получите общий QueryClient через `useQueryClient()` и для каждого id попробуйте достать место из кеша через `queryClient.getQueryData(placesKeys.detail(id))`. Если данные есть — возьмите их.
4. Верните массив `Place[]` (только тех мест, которые удалось взять из кеша).
5. На странице `FavoritesPage` используйте этот hook вместо ручного fetch.

Что мы тренируем:

- доступ к содержимому кеша через `queryClient.getQueryData`;
- использование тех же query keys, что и в основных hooks;
- разделение слоёв: страница не знает про `getQueryData`, она зовёт удобный hook.

Подсказка: места, которые ещё не были загружены в этом сеансе, в кеше отсутствуют — это нормально, hook просто их пропустит. Углубляться в `prefetchQuery` не нужно.

## Задание 2. Индикатор обновления списка

На занятии мы оставили `placeholderData: (previous) => previous` в `usePlacesQuery` — при смене страницы или фильтра предыдущий список не исчезает. Это удобно, но пользователь не понимает, что в данный момент летит запрос.

1. На странице `PlacesPage` достаньте из `usePlacesQuery` поле `isFetching`.
2. Если `isFetching === true`, но `isPending === false` (то есть это «обновление поверх уже отрисованного списка»), покажите над списком неблокирующее сообщение или тонкую полоску с текстом «Обновляем список...».
3. Добавьте простой CSS — никакой анимации не обязательно, достаточно небольшого блока с цветом-акцентом.

Что мы тренируем:

- разницу между `isPending` (первый запрос ещё не завершён) и `isFetching` (идёт ЛЮБОЙ запрос, включая обновление при invalidate);
- умение использовать «фоновое обновление» как полноценное состояние UI, а не как «спиннер на всю страницу».

## Задание 3. Свой собственный staleTime для деталей места (необязательное, сложное)

`usePlaceQuery` сейчас наследует общий `staleTime: 30_000`, который мы задали в `main.tsx`. Допустим, мы хотим, чтобы детали места оставались свежими дольше списка — например, 5 минут — потому что они меняются реже.

1. Откройте `src/features/places/usePlaceQuery.ts`.
2. Добавьте в `useQuery({ ... })` локальную опцию `staleTime: 5 * 60_000`.
3. Проверьте в Devtools:
   - открыли `/places/3` — запись `['places','detail','3']` со status fresh;
   - перешли на другую страницу, потом вернулись через 1 минуту — запроса нет, данные ещё «fresh»;
   - тот же сценарий через 6 минут — данные «stale», TanStack Query тихонько refetch'нет.
4. Подумайте и впишите в `guide.md` ученика (или в комментарий в коде) свой ответ на вопрос: «А что произойдёт, если другой пользователь поменяет это место за эти 5 минут — увидим ли мы изменения сразу?»
5. Дополнительно (по желанию): добавьте кнопку «Обновить» на страницу деталей, которая принудительно делает `placeQuery.refetch()` — это альтернатива `invalidateQueries` для одного места.

Что мы тренируем:

- локальные опции `useQuery`, которые перекрывают `defaultOptions`;
- более точечную настройку кеширования под характер данных;
- ручной `refetch` как один из инструментов «обновить сейчас».

Подсказки:

- общий `staleTime: 30_000` для списка трогать не нужно — список меняется чаще, и 30 секунд для него подходят;
- если перепутаете `staleTime` и `gcTime`, плагин не подсветит, но Devtools покажет: stale-метка не появится в нужный момент.
